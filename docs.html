CRM Distribuidora - Documentación

Documentación completa del sistema CRM para distribuidora de alimentos.

Sistema diseñado para gestionar ventas, logística de entregas, control financiero y comunicación multicanal con clientes del sector HORECA (hoteles, restaurantes, cafeterías), supermercados y tiendas minoristas.

Producto SaaS Multi-Tenant Este CRM está diseñado para venderse a múltiples distribuidoras. Cada cliente (organización) tiene sus propios datos, branding y usuarios, todo dentro de una misma base de datos.

Arquitectura Multi-Tenant

El sistema usa una arquitectura multi-tenant con base de datos compartida. Esto significa que todas las distribuidoras que usen el CRM comparten la misma base de datos y la misma aplicación, pero cada una solo ve sus propios datos.

¿Cómo funciona?

Tabla organizaciones

Cada distribuidora que compra el CRM se registra como una organización. Esta tabla guarda su nombre, logo, colores, configuración y plan de suscripción.

Campo organizacion_id en todas las tablas

Cada registro (cliente, producto, pedido, etc.) tiene un campo organizacion_id que indica a qué distribuidora pertenece. Esto es la clave del aislamiento de datos.

RLS filtra automáticamente

PostgreSQL Row Level Security se encarga de que cada usuario solo vea datos de su organización. No hay forma de acceder a datos de otra org, ni siquiera manipulando la API.

Branding dinámico

Al hacer login, la app carga los datos de la organización (nombre, logo, colores) y los aplica sobre las CSS variables. Cada distribuidora ve "su" CRM personalizado.

Flujo de registro

REGISTRO DE NUEVA DISTRIBUIDORA (owner):
  1. Usuario se registra con metadata: { nombre_organizacion: "Dist. López" }
  2. Trigger automático crea la organización en la tabla
  3. Trigger crea el usuario con rol "owner"
  4. Trigger crea listas de precios por defecto (Tarifa A/B/C)
  5. El owner ya puede loguearse y empezar a cargar datos

INVITAR USUARIO A UNA ORG EXISTENTE:
  1. Owner crea usuario con metadata: { organizacion_id: "uuid-de-la-org", rol: "vendedor" }
  2. Trigger detecta el organizacion_id y lo asigna a esa org
  3. El nuevo usuario entra y ve los datos de la organización

Configuración de Supabase

Crear proyecto en Supabase

Ir a supabase.com, crear una cuenta y un nuevo proyecto. Elegir la región más cercana y anotar la contraseña de la base de datos.

Ir a SQL Editor en el panel de Supabase y ejecutar los 3 archivos en orden:

1. supabase/01_tablas.sql         → Crea 11 tablas + índices + triggers
2. supabase/02_rls_policies.sql   → Configura permisos por rol
3. supabase/03_trigger_nuevo_usuario.sql → Auto-perfil + datos iniciales

Copiar credenciales

Ir a Settings → API y copiar:

  • Project URL → SUPABASE_URL
  • anon/public key → SUPABASE_ANON_KEY

Pegarlas en js/config/supabase.js:

js/config/supabase.js
const SUPABASE_URL = 'https://xxxxx.supabase.co';
const SUPABASE_ANON_KEY = 'eyJhbGci...';

Crear primer usuario (admin)

Ir a Authentication → Users → Add User:

  • Email: tu email
  • Password: tu contraseña
  • Auto Confirm: activado

Luego ir a SQL Editor y actualizar su rol:

UPDATE public.usuarios
SET nombre = 'Tu Nombre', rol = 'admin'
WHERE email = 'tu@email.com';

Los permisos se implementan con dos funciones helper:

SQL
-- Obtener la organización del usuario autenticado
CREATE FUNCTION public.get_user_org_id()
RETURNS UUID AS $$
  SELECT organizacion_id FROM public.usuarios WHERE id = auth.uid();
$$ LANGUAGE sql SECURITY DEFINER STABLE;

-- Obtener el rol del usuario autenticado
CREATE FUNCTION public.get_user_rol()
RETURNS TEXT AS $$
  SELECT rol FROM public.usuarios WHERE id = auth.uid();
$$ LANGUAGE sql SECURITY DEFINER STABLE;

-- Ejemplo de política (multi-tenant + rol)
CREATE POLICY "clientes_select" ON public.clientes
  FOR SELECT TO authenticated
  USING (
    organizacion_id = public.get_user_org_id()  -- PRIMERO: filtro por organización
    AND (
      public.get_user_rol() IN ('owner', 'admin', 'gerente', 'administrativo')
      OR vendedor_asignado_id = auth.uid()      -- SEGUNDO: filtro por rol
    )
  );

Autenticación

El sistema usa Supabase Auth con email y password. El flujo es:

1. Usuario ingresa email + password
2. Supabase Auth valida credenciales
3. Si es correcto → devuelve JWT + session
4. App obtiene perfil del usuario (tabla usuarios)
5. App renderiza la interfaz según su rol
Trigger automático Cuando se crea un usuario en Supabase Auth, automáticamente se crea su registro en la tabla public.usuarios gracias al trigger handle_new_user().

Módulo Pipeline Kanban

js/pages/pipeline.js + css/components/pipeline.css

Tablero Kanban visual que representa el embudo de ventas de la distribuidora. Permite ver de un vistazo en qué estado están todas las negociaciones con clientes y cuánto se espera facturar.

¿Qué es el Pipeline?

El Pipeline (o embudo de ventas) es un tablero con 6 columnas. Cada columna representa una etapa en el proceso de venta. Las tarjetas se mueven de izquierda a derecha a medida que la negociación avanza.

Ejemplo: un vendedor contacta a "Supermercado López" → crea una oportunidad en "Contacto Inicial" → cuando confirma interés la arrastra a "Calificación" → le envía presupuesto → la arrastra a "Presupuesto" → y así hasta que se convierte en "Cliente Activo".

¿Qué es una Oportunidad?

Una oportunidad es un registro que representa un negocio potencial con un cliente. No es un pedido (eso ya existe en el módulo Pedidos), sino la expectativa de venta antes de que se concrete. Contiene:

  • Cliente: Con qué establecimiento se está negociando
  • Valor estimado: Cuánto se espera que compre (ej: ARS 500.000)
  • Probabilidad de cierre: Qué tan probable es que se cierre la venta (ej: 50%)
  • Etapa actual: En qué punto del proceso de venta está
  • Vendedor asignado: Quién lleva esta negociación
  • Próxima acción: Qué hay que hacer y para cuándo (ej: "Enviar presupuesto actualizado" para el 20/02)

Las 6 Etapas del Embudo

Etapa¿Qué significa?Probabilidad por defecto
Contacto Inicial Se hizo el primer contacto con el cliente (llamada, WhatsApp, visita). Todavía no se sabe si hay interés real. 10%
Calificación Se confirmó que el cliente tiene interés, capacidad de compra y encaja con los productos de la distribuidora. 25%
Presupuesto Se le envió una cotización o lista de precios al cliente. Está evaluando. 50%
Negociación El cliente está interesado pero negociando condiciones: precios, plazos de pago, frecuencia de entrega, etc. 70%
Primer Pedido El cliente aceptó y está por hacer (o ya hizo) su primer pedido. Falta concretar logística. 90%
Cliente Activo Venta cerrada. El cliente ya compró y es un cliente activo de la distribuidora. 100%

¿Qué muestra cada tarjeta?

Cada tarjeta en el tablero muestra esta información:

  • Nombre del cliente (arriba) — El establecimiento con el que se negocia
  • Valor estimado (abajo-izquierda) — Cuánto se espera facturar (ej: ARS 500.000)
  • Probabilidad (abajo-centro) — El porcentaje de probabilidad de cierre (ej: 50%)
  • Días en etapa (abajo-derecha, con reloj) — Cuántos días lleva la tarjeta en esa columna sin moverse. Sirve para detectar negociaciones estancadas
  • Vendedor (avatar y nombre) — Quién está a cargo de esta negociación
  • Próxima acción (al pie, con reloj) — Qué hay que hacer próximamente. Si la fecha ya pasó, se muestra en rojo
  • Botones de editar/eliminar — Aparecen al pasar el mouse sobre la tarjeta

¿Qué es el Pronóstico Ponderado?

Es el cálculo más importante del pipeline:

Valor Estimado × Probabilidad = Pronóstico

Ejemplo: Si un cliente tiene un valor estimado de $500.000 y está en etapa "Presupuesto" (probabilidad 50%), el pronóstico es $250.000.

El resumen superior del tablero suma todos los pronósticos de todas las oportunidades. Esto te da una estimación realista de cuánto vas a facturar considerando que no todas las negociaciones se van a cerrar.

Métrica del resumenQué muestra
OportunidadesCantidad total de oportunidades abiertas en el pipeline
Valor TotalSuma de todos los valores estimados (el "mejor caso" si se cierran todas)
Pronóstico PonderadoSuma de (valor × probabilidad) de cada oportunidad. Es la estimación realista de facturación
Con AlertaCantidad de tarjetas que llevan más de 5 días sin moverse (necesitan atención)

Sistema de Alertas Visuales

Las tarjetas cambian de aspecto según cuánto tiempo llevan sin moverse de etapa:

IndicadorCondiciónSignificado
Sin borde especial 0 a 5 días en la etapa Todo normal, la negociación está activa
Borde amarillo Más de 5 días sin moverse Atención: la negociación puede estar enfriándose. Hay que hacer seguimiento
Borde rojo Más de 10 días sin moverse Alerta: riesgo alto de perder la venta. Se necesita acción inmediata

Drag & Drop (Arrastrar y Soltar)

Para mover una oportunidad de etapa:

  1. Hacé click y mantenelo presionado sobre una tarjeta
  2. Arrastrala hasta la columna de la nueva etapa
  3. Soltala

Al soltar, automáticamente:

  • Se actualiza la etapa en Supabase
  • Se reinicia el contador de "días en etapa" a 0
  • Se ajusta la probabilidad al valor por defecto de la nueva etapa
  • Se recalculan los totales y pronósticos del resumen

Nota: la probabilidad se puede editar manualmente después si no coincide con el default.

Al hacer click en "Nueva Oportunidad" se abre un formulario con:

Ejemplo de uso típico

  1. Un vendedor contacta a "Restaurante El Buen Sabor" → Crea oportunidad en "Contacto Inicial" con valor $200.000
  2. El restaurante muestra interés → Arrastra la tarjeta a "Calificación" (prob. sube a 25%)
  3. Le envía presupuesto de lácteos y congelados → Arrastra a "Presupuesto" (prob. 50%)

Durante todo el proceso, el pronóstico ponderado se va ajustando: empezó en $20.000 (200k × 10%) y terminó en $200.000 (200k × 100%).

Sesión 3 — Fase 4 + Bugfixes + Optimizaciones

Bug Fix: Error RLS al crear clientes y productos

Problema: Al intentar crear un cliente o producto aparecía el error "new row violates row-level security policy".

Causa raíz: Los inserts en clientes, productos, pedidos y productos_pedido no incluían el campo organizacion_id. La política RLS de INSERT verifica que organizacion_id = get_user_org_id(), pero al no enviarse el campo era NULL y la validación fallaba.

Solución: Se agregó organizacion_id: window.App.userProfile.organizacion_id en todos los inserts de:

  • js/pages/clientes.js — saveCliente()
  • js/pages/productos.js — saveProducto() y savePreciosLista()
  • js/pages/pedidos.js — savePedido() (tanto pedido como productos_pedido)
  • js/pages/pipeline.js — saveOportunidad() (ya se creó con el fix)

Nota: En los UPDATE se excluye organizacion_id con destructuring (const { organizacion_id, ...updateData } = data) porque no debe cambiar después de creado.

Bug Fix: Dashboard — variable moneda fuera de scope

Problema: En dashboard.js, la variable moneda se declaraba en render() pero se usaba en loadStats(), causando un error de referencia.

Solución: Se movió la declaración de moneda dentro de loadStats(). Además, ahora el dashboard calcula la morosidad real sumando saldo_pendiente de todos los clientes con saldo > 0.

Optimización: Carga lazy de jsPDF

Antes: La librería jsPDF (~200KB) se cargaba en el <head> de index.html en cada página, incluso si el usuario nunca generaba un PDF.

Después: Se removió del <head>. Ahora se carga dinámicamente solo cuando el usuario hace click en "Descargar PDF" en el detalle de un pedido. Se agregó el método _loadJsPDF() en pedidos.js que crea el tag <script> bajo demanda.

Optimización: Preconnect a CDN

Se agregó <link rel="preconnect" href="https://cdn.jsdelivr.net"> en index.html para que el navegador establezca la conexión al CDN de Supabase antes de necesitar el SDK.
